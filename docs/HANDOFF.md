# 유튜브 채널 지식 문서화 PoC — 핸드오프 문서

> 이 문서는 이전 세션에서 논의된 내용을 다른 에이전트(또는 미래의 나 자신)가 이어받아 작업할 수 있도록 고충실도로 기록한 것입니다. 작성일 기준(2026-07-04) 정보이며, 링크된 출처는 문서 하단 참고자료 섹션 참조.

---

## 1. 프로젝트 목표 (확정)

특정 유튜버 한 명이 다루는 콘텐츠의 한 카테고리를 대상으로, 그 사람이 가진 지식을 **개인이 쉽게 습득할 수 있는 통합 지식베이스** 형태로 가공한다.

- **콘텐츠 특성**: 말/음성 중심 (토크, 리뷰, 인터뷰). 화면 정보(코드, 시각 자료)는 중요하지 않음.
- **결과물 지향점**: 영상별 요약이 아니라, 여러 영상에 흩어진 지식을 개념 단위로 통합한 위키형 지식베이스.
- **PoC 규모**: 영상 5~10개 소규모 검증.
- **용도**: 개인 학습용 (상업화 여부 미확정 — 추후 확인 필요, ToS/저작권 리스크 판단에 영향을 줌).

---

## 2. 원래 제안된 파이프라인 (사용자 초안)

1. 유튜브 영상을 다운로드해 메타데이터와 함께 저장
2. 유튜브가 제공하는 스크립트(자막)가 있으면 저장하고 STT 생략
3. 스크립트가 없으면 SOTA STT로 변환
4. 스크립트를 AI로 지식 문서화

## 3. 논의를 통해 도출된 개선 사항

### 3-1. 원본 확보 단계 (기존 1번)
- 말 중심 콘텐츠이므로 **영상 전체가 아닌 오디오만 다운로드**하는 것을 권장 (`yt-dlp -x`). 저장공간/처리속도 이점.
- 메타데이터는 title, description, upload date, chapters 정도로 충분. YouTube Data API v3가 yt-dlp보다 공식적이고 안정적(단, quota 제한 있음).
- **주의(리스크)**: YouTube 다운로드는 ToS상 원칙적으로 비허용. 개인 학습용 PoC는 그레이존이지만, 서비스화·공개 배포 시에는 채널 소유자 허락 또는 정당한 API 경로 검토 필요. → **미해결 이슈로 남겨둠, 사용자에게 재확인 필요.**

### 3-2. 기존 자막 활용 단계 (기존 2번)
- 유튜브 자막은 두 종류로 구분해야 함:
  - **수동 업로드 자막(크리에이터 제작)**: 신뢰도 높음, 그대로 사용 가능
  - **자동생성 자막**: 이미 낮은 품질의 STT 결과물이므로 **신뢰하지 말고 자체 STT로 재처리 권장**
- "스크립트가 제공되는 경우" 판단 기준에 자동생성 자막을 포함시킬지 여부는 **명시적으로 정책화**해야 함 (현재 권장안: 제외).
- 타임스탬프를 같이 저장해두면 이후 근거 검증(원문 대조)에 활용 가능.

### 3-3. STT 단계 (기존 3번)
- 화자가 2인 이상(인터뷰)인 콘텐츠 특성상 **화자 분리(diarization)**가 필요할 가능성 높음.
- 로컬 무료 옵션(Whisper large-v3, faster-whisper, WhisperX)로 먼저 PoC 진행 후, 품질 부족 시 상용 API 비교 권장.
- 2026년 7월 기준 STT 시장 현황(웹 검색 확인):
  - **Deepgram Nova-3**: 스트리밍/실시간 특화, 다국어 코드스위칭 지원(2026년 1월·3월 확장 이후 36개+ 단일언어 지원), 영어권에서는 WER 우수하지만 <cite index="7-1">한국어 등 도메인 특화 정확도는 독립 벤치마크에서 별도 검증 필요</cite>.
  - **OpenAI Whisper (large-v3)**: 셀프호스팅 가능, 배치 처리에 적합, 비용 효율적. 2026년 기준 <cite index="7-1">NVIDIA Canary Qwen 2.5B가 Open ASR 리더보드 WER 1위, GPT-4o Transcribe가 독립 벤치마크 정확도 1위</cite>로 보고되나 이는 주로 영어 기준.
  - **한국어 특화 옵션**: 리턴제로(Return Zero), 네이버 클로바스피치(CLOVA Speech)가 한국어 CER(문자 오류율) 기준 상위권으로 보고됨. 커뮤니티 벤치마크에 따르면 <cite index="12-1">리턴제로 엔진과 리턴제로 Whisper 엔진이 평균 오류율 8% 미만으로 상위권을 차지하고, 그 뒤를 네이버 클로바가 따르고 있음</cite>. 한국어는 <cite index="13-1">교착어 특성상 단어 단위(WER)보다 문자 단위(CER) 평가가 더 적절</cite>하다는 점도 참고.
  - **결론**: 한국어 콘텐츠라면 (1) Whisper large-v3 로컬 실행으로 베이스라인 확보 → (2) 리턴제로 또는 클로바스피치 API와 5~10개 샘플 비교 → 정확도/비용 트레이드오프 판단하는 순서를 권장.

### 3-4. 지식 문서화 단계 (기존 4번) — **가장 중요한 설계 포인트**

목표가 "영상별 요약"이 아니라 "지식 통합"이므로, 서술형 문서를 먼저 만들고 나중에 병합하는 방식은 권장하지 않음 (병합 시 누락/중복/왜곡 발생 위험). 대신 **2단계 구조화 추출 방식**을 제안:

**1단계 — 영상별 "지식 원자 단위(atomic knowledge unit)" 추출**

서술형 요약이 아니라 구조화된 JSON으로 추출:

```json
{
  "concept": "핵심 개념/주제",
  "statement": "영상에서 주장/설명한 핵심 명제 (1~2문장, 원문 근거 기반)",
  "type": "fact | opinion | tip | definition",
  "source_video_id": "...",
  "timestamp": "12:34",
  "quote_evidence": "판단 근거가 된 원문 구절 (짧게, 검증용)"
}
```

- `type` 구분 필수: 말/음성 중심 콘텐츠(토크·리뷰·인터뷰)는 개인 의견과 검증 가능한 사실이 섞여 있으므로 신뢰도 판단 근거로 필요.
- `timestamp`, `quote_evidence`로 원본 역추적 및 할루시네이션 검증 가능하게 설계.

**2단계 — 영상 간 통합 (주제별 병합)**

5~10개 영상에서 나온 지식 원자 단위들을 모아 LLM에게 "같은 concept끼리 클러스터링 → 중복 제거 → 상충 내용 플래깅"을 수행시킴. 구조화된 데이터 병합은 서술형 문서 병합보다 안정적. 결과물은 주제별 마크다운 문서(개념 → 설명 → 출처/타임스탬프 링크)로 만들어 개인 위키(Obsidian 등과 호환) 형태로 완성.

- **PoC 규모(5~10개)에서는 통합 단계 결과를 사람이 직접 검토**하여 할루시네이션·왜곡 여부 확인 권장 (자동 검증 파이프라인은 스케일업 단계에서 고려).

## 4. 수정된 전체 파이프라인 (합의안)

| 단계 | 내용                                                        | 비고                                   |
| ---- | ----------------------------------------------------------- | -------------------------------------- |
| 0    | 영상 선정 (5~10개, 주제 밀도 높은 것 우선)                  |                                        |
| 1    | 오디오 + 메타데이터 다운로드                                | `yt-dlp -x`, ToS 리스크 별도 확인 필요 |
| 2    | 자막 확인 (수동 자막만 신뢰, 자동생성 자막은 재처리)        | 타임스탬프 보존                        |
| 3    | STT (로컬 Whisper 베이스라인 → 한국어 특화 API 비교)        | 화자분리 필요시 WhisperX               |
| 4    | 텍스트 정제 (filler, 광고/인사 구간 제거)                   | 규칙 기반 + LLM 1차 클렌징             |
| 5    | 영상별 지식 원자 단위 추출 (LLM, 구조화 출력)               | type: fact/opinion/tip/definition      |
| 6    | 영상 간 통합 (클러스터링 → 중복제거 → 상충 플래깅 → 문서화) | 사람 검토 필수(PoC 단계)               |

## 5. 용어 설명

| 용어                                       | 설명                                                                                                                                                       |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **STT (Speech-to-Text)**                   | 음성을 텍스트로 변환하는 기술/서비스                                                                                                                       |
| **WER (Word Error Rate)**                  | 단어 단위 오류율. 영어 등 띄어쓰기 기반 언어의 STT 정확도 평가에 주로 사용                                                                                 |
| **CER (Character Error Rate)**             | 문자 단위 오류율. 한국어처럼 교착어(조사·어미가 붙는 언어) 특성상 단어 경계가 모호한 언어에서는 WER보다 CER이 더 적절한 평가 지표로 간주됨                 |
| **Diarization (화자 분리)**                | 오디오에서 "누가 말했는지"를 구분해 화자별로 텍스트를 분리하는 기능. 인터뷰/토크 콘텐츠에서 중요                                                           |
| **지식 원자 단위 (atomic knowledge unit)** | 본 프로젝트에서 정의한 개념: 하나의 검증 가능한 최소 지식 조각(개념+명제+근거+출처)을 구조화된 데이터로 표현한 것. 서술형 요약과 달리 병합·중복제거가 쉬움 |
| **할루시네이션 (hallucination)**           | LLM이 원본에 없는 내용을 사실인 것처럼 생성하는 현상. 본 프로젝트에서는 timestamp/quote_evidence로 역추적 가능하게 설계해 완화                             |
| **yt-dlp**                                 | 유튜브 등 동영상 사이트에서 영상/오디오/자막/메타데이터를 다운로드하는 오픈소스 커맨드라인 도구                                                            |
| **Whisper**                                | OpenAI가 공개한 오픈소스 STT 모델. 셀프호스팅 가능, large-v3가 최신 안정 버전                                                                              |
| **ToS (Terms of Service)**                 | 서비스 이용약관. 유튜브 영상 다운로드는 원칙적으로 ToS 위반 소지가 있어 용도(개인/상업)에 따라 리스크 판단 필요                                            |

## 6. 미해결 이슈 / 다음 세션에서 확인할 것

1. **ToS/저작권 리스크**: 이 PoC가 순수 개인 학습용인지, 추후 서비스화·공유 가능성이 있는지 명확히 하지 않음. 이에 따라 다운로드 방식과 데이터 보관 정책이 달라짐.
2. **STT 도구 실제 비교**: Whisper large-v3(로컬) vs 리턴제로 vs 클로바스피치를 실제 샘플 5~10개로 CER 비교하는 실험 미실시. 다음 단계로 추천.
3. **5단계(지식 원자 추출) 프롬프트 설계**: 구체적인 LLM 프롬프트/스키마 검증(예: type 분류 정확도, concept 명명 일관성)이 아직 설계되지 않음.
4. **6단계(통합) 알고리즘**: concept 클러스터링을 순수 LLM에 맡길지, 임베딩 기반 유사도 클러스터링 후 LLM으로 정제할지 결정 필요. 5~10개 규모에서는 LLM만으로 충분할 가능성 높으나, 스케일업 시 재검토 필요.
5. **대상 채널/카테고리**: 아직 구체적인 유튜버/카테고리가 특정되지 않음.
